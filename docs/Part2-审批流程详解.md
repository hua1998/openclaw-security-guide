## 一、审批流*- 其实很简*
### 1.1 什么情况需要审批？

#### *无需审批 (70% 操作)

**场景**: 日常查询、只读操作、测试环*
**示例**:
```
用户*帮我查看当前目录的文*
AI:  直接执行 ls -la
耗时* *审批：无需
```

**条件**:
- *白名单命*(ls, cat, grep *
- *只读操作
- *测试环境
- *可信用户 (信任分≥80)

**自动审批**: 70%

---

#### ⚠️ 自己确认 (20% 操作)

**场景**: 安装包、文件写入、普通变*
**示例**:
```
用户*安装 lodash 依赖"

AI 显示确认卡片:
┌─────────────────────────────**操作确认                     **命令：npm install lodash     **风险：*中等                **影响：当前项*              **                             **[*确认]  [*取消]         *└─────────────────────────────*
用户点击"确认" *AI 执行
耗时*0 *```

**条件**:
- ⚠️ 灰名单命*(npm install, pip install *
- ⚠️ 文件写入操作
- ⚠️ 普通风险评*(30-70 *

**自动审批**: 用户自己确认，无需他人

---

#### 🔴 管理员审*(10% 操作)

**场景**: 生产部署、数据库变更、基础设施

**示例**:

**条件**:
- 🔴 生产环境操作
- 🔴 数据库变*- 🔴 基础设施变更

### 1.4 自动化减少人工审*
#### 自动审批规则

```yaml
自动通过:
  # 规则 1: 测试环境操作
  - 条件：environment == "test"
    动作：自动批*  
  # 规则 2: 低风*+ 可信用户
  - 条件：risk_score < 30 AND user_trust >= 80
    动作：自动批*  
  # 规则 3: 重复请求
  - 条件：same_request_within(1h) AND approved_before
    动作：自动批*  
  # 规则 4: 紧急故*  - 条件：severity == "P0" AND user_role == "oncall"
    动作：自动批*(事后通知主管)

预期效果:
  自动审批率：>70%
  人工审批*30%
  平均响应时间*1 分钟
```

#### 实际效果对比

**实施**:
```
总请求：100 *人工审批*0 *(80%)
自动审批*0 *(20%)
平均响应*5 分钟
```

**实施**:
```
总请求：100 *人工审批*5 *(25%)
自动审批*5 *(75%)
平均响应* 分钟
```

**改进**:
- 人工审批减少*8%
- 响应时间提升*3%
- 用户满意度：4.5/5

### 1.7 常见问题

**Q1: 审批人不在怎么办？**
```
A: 配置备份审批*   - 第一审批人：团队主管
   - 第二审批人：运维主管
   - 超时升级*0 分钟未审批，自动通知第二审批*```

**Q2: 紧急故障需要立即处理怎么办？**
```
A: 启用紧急通道
   - On-call 人员可直接处*   - 事后补审批流*   - 自动通知主管
```

**Q3: 审批太多处理不过来怎么办？**
```
A: 优化自动审批规则
   - 分析审批数据，识别可自动化的场景
   - 增加自动审批规则
   - 目标：自动审批率>70%
```

**Q4: 如何避免审批流于形式**
```
A: 审批质量监控
   - 统计审批通过*   - 识别"无脑批准"的审批人
   - 定期培训审批*   - 设置审批准确率指*```

### 2.1 第一步：配置监控指标

**文件**: `config/monitoring.yaml`

```yaml
# 复制粘贴这个配置就行

metrics:
  # 安全指标
  security_incidents:
    target: 0
    alert_threshold: 1
  
  audit_log_completeness:
    target: 100%
    alert_threshold: 95%
  
  policy_violations:
    target: <10/*    alert_threshold: 20/*  
  # 效率指标
  auto_approval_rate:
    target: 70%
    alert_threshold: 50%
  
  false_positive_rate:
    target: <1%
    alert_threshold: 5%
  
  system_availability:
    target: 99.9%
    alert_threshold: 99%
  
  # 响应指标
  avg_response_time:
    target: 10m
    alert_threshold: 30m
  
  approval_timeout_rate:
    target: <5%
    alert_threshold: 10%
```

**实施时间**: 10 分钟

### 2.2 第二步：配置告警通知

**文件**: `config/alerts.yaml`

```yaml
# 飞书通知 (复制修改 webhook 地址)
feishu_webhook: "https://open.feishu.cn/open-apis/bot/v2/hook/xxx"

# 短信通知 (P0 告警)
sms_provider: "aliyun"
sms_config:
  access_key: "xxx"
  phone_numbers:
    - "13800138000"  # 安全负责*    - "13900139000"  # 运维负责*
# 告警规则
alerts:
  # P0 紧急告*  - name: 安全事件
    condition: security_incidents > 0
    notify:
      - feishu_webhook
      - sms
    severity: P0
  
  # P1 高优先级告警
  - name: 审批超时
    condition: pending_approval > 30m
    notify:
      - feishu_webhook
    severity: P1
  
  # P2 中优先级告警
  - name: 自动审批率低
    condition: auto_approval_rate < 50%
    notify:
      - feishu_webhook
    severity: P2
  
  # P3 低优先级告警
  - name: 误伤率高
    condition: false_positive_rate > 5%
    notify:
      - feishu_webhook
    severity: P3
```

**实施时间**: 10 分钟

### 2.6 常见问题

**Q: 没有 Grafana 怎么办？**
```
A: 使用飞书多维表格，或者直接用 OpenClaw 自带*Web 控制*   推荐：OpenClaw 控制*(开箱即*
```

**Q: 告警太多怎么办？**
```
A: 调整告警阈值，或者设置告警聚*   配置*   alert_aggregation:
     enabled: true
     window: 5m
     max_alerts: 10
```

**Q: 告警没人处理怎么办？**
```
A: 设置告警升级机制
   配置*   escalation:
     timeout: 30m
     escalate_to: backup_approver
     final_escalate: cto
```
